Section 1

The Product

Vaani Auto lets the driver and passengers talk to the car naturally, in any of 22 Indian languages including mixed-language speech, and have the car respond by voice and by action: adjusting the cabin, navigating, placing calls and reading messages, answering questions about the vehicle and the world, and narrating what the safety systems are seeing.

It runs on the vehicle's Qualcomm Snapdragon cockpit compute (Hexagon NPU) via Sarvam Edge, with an India-hosted cloud used only as overflow for Live Data and complex, Long tail queries. Roughly 90% of interactions never leave the vehicle, which suits patchy highway connectivity, keeps data in the car for DPDP compliance, and carries zero per-query cloud cost. The stack aligns with Sarvam's real partnerships: Qualcomm for chip optimization, Bosch for automotive assistants.

What it is not: it is never part of the safety-critical ADAS control loop. It reads validated safety state through a one-way interface and speaks it; it does not steer, brake, or actuate any ASIL-rated function. Every design decision follows from this boundary.

Each component in Figure 1 has one narrow job. None of their responsibilities overlap, which is what keeps the control path auditable: at any point in the flow, there is exactly one component that could have taken an action, and it is always the Deterministic Command Executor.

Section 2

Design Principles

Six commitments that shape every decision in this document.

  • Offline-first. Core interaction works with zero connectivity; cloud is an enhancement, never a dependency.
  • Latency-bounded. On-device actions are near-instant; safety announcements are immediate and pre-empt all other speech.
  • Safety-isolated. The assistant lives in the cockpit domain, reading the ASIL-D domain through a one-way validated interface only.
  • Privacy by default. In-cabin audio and camera video are processed on-device and never leave the vehicle.
  • Indian-language-native. All 22 languages run on-device, so language quality never degrades offline.
  • Untrusted input by default. Everything the system reads is content to interpret, never an instruction to obey; actions pass only through a fixed, validated command schema.
Section 3

Architecture & Routing

Three tiers: the on-vehicle edge handles everything latency-sensitive, safety-relevant, privacy-sensitive, or needed offline; the India-hosted cloud (Sarvam 30B/105B, live data) is invoked only as overflow; and a lightweight on-device Policy Router decides between them per utterance.

Figure 2. The per-utterance routing policy. Vehicle control and safety are always edge; fresh data and heavy reasoning go to cloud when connected; offline, the assistant says so plainly and offers the on-device alternative.

The Intent Classifier itself is a small multilingual text-encoder model with a classification head, not a generative model and not the Voice Assistant LLM. It embeds the driver's utterance and classifies it against the fixed intent schema, or flags it as informational, producing a confidence score that feeds directly into the router: low confidence is what triggers cloud overflow or a confirm-before-act prompt. It deliberately reuses the same class of on-device embedding model that Section 4's retrieval uses, so the vehicle carries one shared embedding stack doing double duty rather than two separate models competing for the same NPU budget.

Why an encoder, not another LLM. Classification against a fixed, finite label set is a bounded, testable operation: build a per-language, per-intent eval set and measure accuracy directly. Generation isn't bounded the same way. Keeping intent classification here, architecturally separate from the Voice Assistant LLM, keeps the highest-stakes decision in the pipeline, is this a command, measurable rather than probabilistic.
Section 4

On-Device Retrieval (RAG)

On-device retrieval, retrieval-augmented generation in full, is what lets a sub-1GB model answer manual, destination, and contact questions accurately without memorizing them and without a network. Baking this content into model weights doesn't work: the vehicle manual runs to hundreds of pages, contacts and saved destinations are per-owner and change constantly, and a small on-device model has no room to memorize any of it reliably. Retrieval keeps the model's job narrow: read a handful of relevant passages, then compose an answer strictly from what's in front of it.

Figure 3. On-device retrieval grounds informational answers in local content: vehicle manual, saved destinations and contacts, cached map tiles, and current vehicle state, indexed once and searched fresh per turn. Control commands never touch this index; they run through the fixed intent schema straight to the Deterministic Executor. Gold dashed is local context and retrieved data, solid dark is on-device flow, purple is the LLM.

Indexing happens once, not per query. The vehicle manual, saved destinations and contacts, cached map tiles, and a snapshot of current vehicle state are chunked and indexed on-device at OTA update time, with contacts and destinations refreshed as they change. A driver's question is embedded on-device, matched against that index by similarity search, and the top few passages are handed to the LLM to compose a spoken answer from, not to reason without bounds.

Retrieval never touches the control plane. A question about a dashboard warning light gets an answer grounded in retrieved manual text. A command like "roll down the rear window" never touches the index at all; it goes straight from the Intent Classifier through the fixed schema to the Deterministic Executor (see Architecture & Routing). The same separation that keeps hallucinations and prompt injection out of the control plane keeps retrieval out of it too.

The on-device embedding model doing the similarity search here is the same one behind the Intent Classifier's confidence score (see Architecture & Routing): one shared embedding stack, two jobs, rather than two separate models on an already-tight NPU budget.

Two things here are assumptions, not measurements, and are logged as open items rather than settled facts. Retrieval overhead is currently covered by the same "Grounding + executor, 40ms" line in the latency budget (see Latency) that existed before this section did, not independently re-measured for an actual embedding and similarity-search step. And even with intent classification and retrieval sharing one embedding model, its footprint isn't yet broken out from the sub-1GB speech-stack figure, so total NPU memory pressure across STT, translation, TTS, and the shared embedding model together is still an assumption, not a bill of materials.

Section 6

Safety, Privacy & Trust

Figure 4. Domain isolation: detection lives in the certified ASIL-D domain; the assistant reads validated state through a one-way interface and narrates it. No write path exists, and the assistant has no actuation capability, so it cannot become a remote actuation vector either.

One property does most of the work here, twice over: the control plane (what actions are permitted) is separated from the content plane (the language the model reads). Vehicle commands execute only through a fixed, validated intent schema, so neither a hallucinated instruction nor one smuggled into a read-aloud message, place name, or search result can cause an action.

  • Confidence gating. Below threshold, confirm or decline; never guess on a control or safety action.
  • Grounded and honest. Answers are grounded on retrieved data (see On-Device Retrieval); when uncertain or offline, the assistant says so plainly. It never fabricates.
  • Priority arbitration. Safety announcements always pre-empt lower-priority speech.
  • Data locality. Audio and cabin video are processed on-device; cloud overflow carries de-identified text only, over encrypted channels, to India-hosted infrastructure. DPDP-ready with attestation and an audit trail.
  • Secure updates. Signed, attested OTA model updates with rollback.
Section 7

Latency

The number the driver feels is voice-to-voice: end of utterance to first spoken syllable. STT and TTS both stream, so each stage counts by time to first byte. The edge path lands around a second. Cloud overflow now budgets roughly 3s end to end, reserved for non-urgent queries; the moment the router commits to a cloud call it fires an immediate spoken acknowledgement, "checking this over the internet", so the wait reads as active rather than silent.

Figure 5. Stage-by-stage budget for both paths. Endpointing and TTS first byte bracket every turn; cloud-routed queries get an immediate spoken acknowledgement the moment the router commits to cloud (~750ms), and the assistant still times out into the graceful spoken fallback at 3 to 4s, now close to the cloud path's typical latency.

Open questions on sourcing

In addition to the three architectural open questions named in Section 8, these are open at the level of the numbers themselves:

  • Cloud latency margin against fallback. Typical cloud response (~3.05s) now sits right at the start of the 3-4s fallback timeout window. Either the threshold needs revisiting, cloud needs a lower-latency deployment path, or this margin is accepted as-is with the immediate acknowledgement doing the work of masking it.
  • Pipelining vs. serial summation. All waterfall totals are naive sums of standalone benchmarks. A real streaming implementation could overlap stages and lower actual latency below what's stated here; no integrated pipeline benchmark exists to confirm this.
  • Several waterfall components remain unbenchmarked budget estimates: VAD endpoint, intent/routing, grounding/executor, network send, and cloud return+TTS. Lowest-confidence numbers in the latency story.
  • Bulbul TTS's 260ms figure is a phone benchmark (Galaxy S25 Ultra), not automotive silicon. Directionally fine since both run on Hexagon NPU class hardware, but not a like-for-like number.
  • The ~90% on-device resolution share is the weakest-sourced headline number in this document. No internal measurement and no external benchmark back it, only a directional argument from the feature catalog's own placement mix.
  • The on-device retrieval and Intent Classifier sections both depend on an embedding model whose footprint and latency aren't separately budgeted. If asked for a bill of materials on top of Saaras/Mayura/Bulbul, there isn't one yet.
