Section 1

The Product

Vaani Auto lets the driver and passengers talk to the car naturally, in any of 22 Indian languages including mixed-language speech, and have the car respond by voice and by action: adjusting the cabin, navigating, placing calls and reading messages, answering questions about the vehicle and the world, and narrating what the safety systems are seeing.

It runs on the vehicle's Qualcomm Snapdragon cockpit compute (Hexagon NPU) via Sarvam Edge, with an India-hosted cloud used only as overflow for Live Data and complex, Long tail queries. Roughly 90% of interactions never leave the vehicle, which suits patchy highway connectivity, keeps data in the car for DPDP compliance, and carries zero per-query cloud cost. The stack aligns with Sarvam's real partnerships: Qualcomm for chip optimization, Bosch for automotive assistants.

What it is not: it is never part of the safety-critical ADAS control loop. It reads validated safety state through a one-way interface and speaks it; it does not steer, brake, or actuate any ASIL-rated function. Every design decision follows from this boundary.

Section 2

Design Principles

Six commitments that shape every decision in this document.

  • Offline-first. Core interaction works with zero connectivity; cloud is an enhancement, never a dependency.
  • Latency-bounded. On-device actions fire before the spoken confirmation finishes, so the action lands faster than the ~1s voice-to-voice figure; safety announcements are immediate and pre-empt all other speech.
  • Safety-isolated. The assistant lives in the cockpit domain, reading the ASIL-D domain through a one-way validated interface only.
  • Privacy by default. In-cabin audio and camera video are processed on-device and never leave the vehicle.
  • Indian-language-native. All 22 languages run on-device, so language quality never degrades offline.
  • Untrusted input by default. Everything the system reads is content to interpret, never an instruction to obey; actions pass only through a fixed, validated command schema.
Section 3

Architecture & Routing

Three tiers: the on-vehicle edge handles everything latency-sensitive, safety-relevant, privacy-sensitive, or needed offline; the India-hosted cloud (Sarvam 30B/105B, live data) is invoked only as overflow; and a lightweight on-device Policy Router decides between them per utterance.

Figure 2. The per-utterance routing policy. Vehicle control and safety are always edge; fresh data and heavy reasoning go to cloud when connected; offline, the assistant says so plainly and offers the on-device alternative.

The Intent Classifier itself is a small multilingual text-encoder model with a classification head, not a generative model and not the Voice Assistant LLM. It embeds the driver's utterance and classifies it against the fixed intent schema, or flags it as informational, producing a confidence score that feeds directly into the router: low confidence is what triggers cloud overflow or a confirm-before-act prompt. It deliberately reuses the same class of on-device embedding model that Section 4's retrieval uses, so the vehicle carries one shared embedding stack doing double duty rather than two separate models competing for the same NPU budget. When the classifier cannot cleanly separate a command from a comment, an utterance like "it's hot in here", the tie always resolves toward confirm-before-act on the edge: the assistant asks, it never silently actuates, and it never ships an ambiguous control candidate to the cloud.

Why an encoder, not another LLM. Classification against a fixed, finite label set is a bounded, testable operation: build a per-language, per-intent eval set and measure accuracy directly. Generation isn't bounded the same way. Keeping intent classification here, architecturally separate from the Voice Assistant LLM, keeps the highest-stakes decision in the pipeline, is this a command, measurable rather than probabilistic.

The "fixed intent schema" is a concrete object, not a figure of speech. Every utterance the classifier tags as a control command is matched against it before anything actuates: the schema enumerates exactly which intents exist, what arguments each accepts, and how each is gated. An utterance that matches no intent, or whose arguments fall outside the allowed set, is refused rather than guessed at, which is what turns "the system cannot invent an action" into a structural property instead of a hope. A representative slice:

// Fixed intent schema, matched by the Deterministic Command Executor.
// A control command runs only if it matches an intent AND its slots validate.
{
  // Comfort controls: bounded enums / numeric ranges, no free-text slot
  "climate.set": {
    "slots": {
      "zone":   ["driver", "passenger", "rear", "all"],
      "temp_c": { "type": "int", "range": [16, 30] }
    },
    "gating": "none"
  },
  "media.volume": {
    "slots": {
      "action": ["up", "down", "set", "mute"],
      "level":  { "type": "int", "range": [0, 40] }
    },
    "gating": "none"
  },

  // Motion-affecting: speed- and confirm-gated at the executor
  "window.move": {
    "slots": {
      "window":     ["driver", "passenger", "rear_left", "rear_right", "all"],
      "action":     ["up", "down"],
      "extent_pct": { "type": "int", "range": [0, 100] }
    },
    "gating": ["speed_gated", "confirm"]
  },

  // Parameterized invocations: entity resolved via a structured store, then confirmed
  "navigation.start": {
    "slots": {
      "destination": { "type": "entity", "resolve_via": "saved_destinations | poi_lookup" }
    },
    "gating": ["confirm"]
  },
  "call.place": {
    "slots": {
      "contact": { "type": "entity", "resolve_via": "contact_store" }
    },
    "gating": ["confirm"]
  },
  "message.send": {
    "slots": {
      "contact": { "type": "entity", "resolve_via": "contact_store" },
      "body":    { "type": "string", "source": "dictated_only" }
    },
    "gating": ["confirm", "read_back_before_send"]
  }
}

Two properties fall straight out of the shape. Comfort intents take only bounded enums or numeric ranges, so there is no free-text field for an injected instruction to hide in. The motion-affecting and parameterized intents each carry explicit gating: a speed check for windows and seats, a spoken confirmation before acting, and, for a dictated reply, a read-back before it sends. Entity arguments (a contact, a destination) resolve through direct structured lookups, never through the retrieval index of Section 4, so a command's target can never come from free-text similarity search. The list here is a representative slice; the shipping schema also covers media playback, seats, and climate fan control on the same pattern.

Section 4

On-Device Retrieval (RAG)

On-device retrieval, retrieval-augmented generation in full, grounds the assistant's informational answers in local content held in an On-Device Index. Four sources feed that index. The vehicle manual contributes owner-manual text and dashboard-warning guidance, chunked for retrieval. Saved destinations and contacts contribute the driver's own places and people. Cached map tiles contribute offline map data and saved routes. And a rolling snapshot of vehicle state contributes the current trip, cabin settings, and diagnostic readings. All four are chunked, embedded, and indexed on the vehicle.

Figure 3. On-device retrieval grounds informational answers in local content: the vehicle manual, saved destinations and contacts, cached map tiles, and current vehicle state are chunked and indexed once into a standing On-Device Index the assistant can consult on any informational turn. Per turn, the driver's question is embedded on-device, matched against the index by similarity search, and the top passages are handed to the LLM, which composes a spoken answer only from that retrieved text. Gold dashed is local context and retrieved data, solid dark is on-device flow, purple is the LLM.

Indexing happens once, not per query: the four sources are chunked and indexed on-device at OTA update time, with contacts and destinations refreshed as they change. On each informational turn, the driver's question is embedded on-device, matched against the index by similarity search, and the top few passages are handed to the Voice Assistant (LLM), a compact on-device model, which composes a spoken answer strictly from that retrieved text rather than reasoning without bounds. Because the index is always resident on the vehicle, this grounding costs no network round trip.

Concretely, these are the kinds of questions the On-Device Index answers, each grounded in a different source and each answerable with zero connectivity:

  • "What does this flashing orange engine light mean, and is it safe to keep driving?" grounded in the vehicle manual and warning guidance.
  • "How do I turn on the rear defogger?" grounded in the vehicle manual, retrieved as a how-to passage rather than triggering an action.
  • "What's the address I saved for the office?" grounded in saved destinations, read back as information, distinct from the command "take me to the office".
  • "Is there a fuel stop on my saved route to Lonavala?" grounded in cached map tiles and saved routes, so it still answers in a dead zone.
  • "What's my tyre pressure, and is that in the normal range?" grounded in the vehicle-state snapshot for the reading and the manual for the safe range, one answer drawn from two sources.

The on-device embedding model doing the similarity search is the same one behind the Intent Classifier's confidence score (see Architecture & Routing): one shared embedding stack, two jobs, rather than two separate models on an already-tight NPU budget.

Section 6

Safety, Privacy & Trust

Figure 4. Domain isolation. Inside the certified safety domain, two kinds of function live side by side: driver-facing warnings (forward collision, lane departure, blind-spot, and the driver-monitoring alerts), which the assistant reads through a one-way interface and speaks, and self-actuating control interventions (adaptive cruise, automatic emergency braking), which stay entirely inside the ASIL-D domain and never feed the assistant. No write path exists back into the domain, and the assistant has no actuation capability, so it cannot become a remote actuation vector either. The Safety Narrator shown here is the arbitration function that sits inside Figure 1's response stage, expanded at this zoom level.

A word on the "ASIL-D" label. It names the safety rating of the strictest functions in that domain, the self-actuating interventions like automatic emergency braking, not every box inside it; driver monitoring and the driver-facing warnings are each certified at their own levels. The label simply marks where the highest-rated functions live and the isolation they demand of the assistant: Vaani reads the validated warnings and speaks them, and it neither triggers nor rides inside any of the control functions.

One property does most of the work here, twice over: the control plane (what actions are permitted) is separated from the content plane (the language the model reads). Vehicle commands execute only through a fixed, validated intent schema, so neither a hallucinated instruction nor one smuggled into a read-aloud message, place name, or search result can cause an action.

  • Confidence gating. Below threshold, confirm or decline; never guess on a control or safety action.
  • Actuation gating. The executor's schema covers cockpit comfort and convenience functions only. Motion-affecting or pinch-risk actuations, windows and seats, are speed-gated and confirm-gated; nothing in the schema can reach an ASIL-rated function.
  • Grounded and honest. Answers are grounded on retrieved data (see On-Device Retrieval); when uncertain or offline, the assistant says so plainly. It never fabricates.
  • Priority arbitration. Owned by the Safety Narrator (Figure 4; inside Figure 1's response stage): safety announcements always pre-empt lower-priority speech, and nothing can pre-empt them.
  • Data locality. Audio and cabin video are processed on-device; cloud overflow carries de-identified text, over encrypted channels, to India-hosted infrastructure. One deliberate exception: live-data lookups (traffic, POI, fuel and EV) attach coarse, query-scoped location, used ephemerally, never stored, and never linked to an identity, because a nearby search is impossible without it. DPDP-ready with attestation and an audit trail.
  • Secure updates. Signed, attested OTA model updates with rollback.
Section 7

Latency

The number the driver feels is voice-to-voice: end of utterance to first spoken syllable. STT and TTS both stream, so each stage counts by time to first byte. The edge path lands around a second. Cloud overflow now budgets roughly 3.2s end to end, reserved for non-urgent queries; the moment the router commits to a cloud call it fires an immediate spoken acknowledgement, "checking this over the internet", so the wait reads as active rather than silent.

The waterfall's edge path is drawn for the command case. Informational edge turns, a manual question, for example, add one stage the chart does not show: the on-device LLM composing the grounded answer, budgeted at 200 to 450ms to first token. That is what stretches manual Q&A to the 1 to 1.5s quoted in the feature catalog, and it is a budget estimate that stays open until the answer-composition model is selected (see Appendix).

The fallback is a delayed-response update, not a hard cut-off, and its clock starts at the acknowledgement, not at the end of the driver's utterance. Once the assistant has said "checking this over the internet", a typical cloud answer lands about 2 to 2.5s later, so most cloud turns finish before the fallback is ever in play. If four seconds pass after that acknowledgement with no answer, Vaani says so plainly, something like "this is taking longer than usual, sit tight while I fetch it", and keeps waiting; it never fabricates an answer and never silently gives up. The remaining open risk is the tail: typical cloud latency is comfortable, but no p95 or p99 figure exists yet to size how often the delayed-response update actually fires.

Figure 5. Stage-by-stage budget for both paths, drawn for the command case on the edge; informational edge turns add an on-device LLM generation stage of 200 to 450ms (see text). The cloud response leg is a network return plus the same local Bulbul TTS used on the edge, for a typical ~3.21s to first audio. Cloud-routed queries get an immediate spoken acknowledgement the moment the router commits (~750ms commit; first ack audio ~1.0s); the fallback is a delayed-response update timed from that acknowledgement, speaking if no answer lands within 4s of it.

Open questions on sourcing

These are open at the level of the numbers themselves:

  • Cloud latency tail against the delayed-response update. The update is timed from the spoken acknowledgement and fires at ack + 4s; typical cloud latency (~3.21s, roughly 2 to 2.5s after the ack) lands well inside that window. What remains open is the tail: no p95 or p99 cloud latency figure exists, and the tail, not the median, is what determines how often the delayed-response update fires.
  • Pipelining vs. serial summation. All waterfall totals are naive sums of standalone benchmarks. A real streaming implementation could overlap stages and lower actual latency below what's stated here; no integrated pipeline benchmark exists to confirm this.
  • Several waterfall components remain unbenchmarked budget estimates: VAD endpoint, intent/routing, grounding/executor, network send, and cloud return+TTS. Lowest-confidence numbers in the latency story.
  • Bulbul TTS's 260ms figure is a phone benchmark (Galaxy S25 Ultra), not automotive silicon. Directionally fine since both run on Hexagon NPU class hardware, but not a like-for-like number.
  • The ~90% on-device resolution share is the weakest-sourced headline number in this document. No internal measurement and no external benchmark back it, only a directional argument from the feature catalog's own placement mix.
  • The bill of materials on top of Saaras/Mayura/Bulbul now exists as budget lines, but nothing in it is benchmarked. The shared embedding model and the 1-2B class answer-composition LM both carry footprint and latency figures derived from arithmetic and inference, not measurement, and the LM's model family is still an open selection.
